# Angular Signals Are Here — And They Change How You Manage State

Angular 17 shipped Signals as a stable feature, and after using them on a real project for a few months, I can say they genuinely reduce the amount of code you write for managing UI state. Here's a practical overview.

## Why Signals, and Why Now?

Angular's default change detection has always worked — but it required you to understand zones, `OnPush`, and RxJS subscriptions to avoid performance pitfalls. Signals introduce a fundamentally different model: values that Angular can track automatically, without any of that ceremony.

For most local component state, the RxJS + `async` pipe pattern was always a bit heavy. Signals fix that.

## The Basics in Under 5 Minutes

Create a Signal:

```typescript
import { signal } from '@angular/core';

const count = signal(0);
```

Read it (call it like a function):

```typescript
console.log(count()); // 0
```

Update it:

```typescript
count.set(5);
count.update(prev => prev + 1);
```

Derive a new value from it:

```typescript
import { computed } from '@angular/core';

const doubleCount = computed(() => count() * 2);
```

That's it. Angular handles the rest.

## A Real Component Example

Here's a counter component. Compare this to the equivalent BehaviorSubject approach and you'll notice the boilerplate reduction immediately:

```typescript
@Component({
  selector: 'app-counter',
  template: `
    <p>Count: {{ count() }}</p>
    <button (click)="increment()">+1</button>
  `,
})
export class CounterComponent {
  count = signal(0);

  increment() {
    this.count.update(v => v + 1);
  }
}
```

No `Subject`, no `async` pipe, no `ngOnDestroy`, no subscription to unsubscribe from.

## When NOT to Replace RxJS

Signals are not a RxJS replacement for everything. HTTP requests, event streams, debouncing, switchMap — RxJS is still the right tool for those. But for simple state that drives the UI, Signals win.

The `toSignal()` bridge lets you convert Observables to Signals when you need both worlds:

```typescript
import { toSignal } from '@angular/core/rxjs-interop';

const data = toSignal(this.http.get('/api/items'), { initialValue: [] });
```

## Key Takeaways

- Signals work without `OnPush` or zone configuration
- `computed()` signals are memoized and only recalculate when needed
- Templates read Signals by calling them: `{{ count() }}`
- RxJS and Signals interoperate cleanly via `toSignal()`
- Best for: local UI state, derived values, simple shared state in services

## Should You Migrate Today?

New code? Yes, use Signals where it makes sense. Existing codebases? Migrate incrementally — start with the simplest BehaviorSubject usages and work outward.

The Angular team has committed to Signals as the long-term direction, so this isn't a temporary feature.
